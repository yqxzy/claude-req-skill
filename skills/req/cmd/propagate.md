# 变更联动更新

当用户消息中出现以下信号时自动触发（无需 /req 命令）：
- 「新增一个变更点」「追加一条变更」
- 「xxx 改为 xxx」「xxx 变更为 xxx」「把 xxx 修改成 xxx」
- 「删除 xxx 变更」「去掉 xxx 功能」
- 「变更点有调整」「需求有变化」

---

**执行步骤**：

**Step A：更新变更点**
1. 读取 `01_changes.md`
2. 将用户描述的变更追加或修改到对应分类中
3. 写入文件，展示变更后的完整变更清单

**Step B：影响评估**
读取 `_state.md`，对每个已完成（打勾）的步骤做一行判断：

| 步骤 | 评估逻辑 |
|------|---------|
| Step3 代码调研 | 新变更是否涉及未调研的模块？ |
| Step4 方案设计 | 新变更是否超出当前方案的覆盖范围？ |
| Step5 测试Case | 新变更是否需要新的测试场景？ |
| Step6 流程图 | 新变更是否改变了流程中的参与者或分支？ |
| Step7 类图 | 新变更是否引入新类或修改已有类的接口？ |

**Step C：自动更新受影响项**

对判断为「需要更新」的已完成步骤：
- **Step5 测试Case**：追加新 Case 到 `04_testcases.md`，TC# 续接上一个编号
- **Step6 流程图**：读取 `05_flowchart.puml`，修改受影响的部分后写回
- **Step7 类图**：读取 `06_classdiagram.puml`，修改受影响的部分后写回
- **Step3 代码调研 / Step4 方案设计**：只标注需要复查，不自动修改（内容复杂，需人工判断）

**Step D：输出更新报告**

```
📝 变更联动更新完成

变更内容：{本次新增/修改的变更描述}

已自动更新：
  ✅ 01_changes.md — 变更清单已更新
  ✅ 04_testcases.md — 新增 T0N：{场景描述}
  ✅ 05_flowchart.puml — {修改了什么}

需人工复查（内容复杂，未自动修改）：
  ⚠️ Step4 方案设计 — 新变更可能超出当前方案范围，建议执行 /req solutions 重新确认
  
未受影响：
  ✓ Step3 代码调研
  ✓ Step7 类图
```
