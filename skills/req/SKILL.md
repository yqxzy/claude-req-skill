# 需求开发全流程管理助手 (req)

当用户执行 `/req <命令>` 时，按照以下规则工作。

## 核心原则

- **每步只加载必要上下文**：不同阶段读取不同文件组合，避免上下文膨胀
- **用户触发，不自动推进**：每个步骤由用户主动发起，完成后等待用户下一指令
- **结构化落地**：每步产出物保存到对应文件，作为后续步骤的输入
- **当前需求由 `_state.md` 追踪**：所有命令默认作用于当前激活的需求

---

## 文件结构约定

每个需求的文件存放在 `~/.claude/requirements/{req-id}-{req-name}/`：

```
00_background.md      # Step 1: 背景信息
01_changes.md         # Step 2: 变更点
02_codebase.md        # Step 3: 现有代码分析
03_solutions.md       # Step 4: 多方案评估
04_flowchart.mmd      # Step 5: 执行流程图（Mermaid）
05_classdiagram.mmd   # Step 6: 核心类图（Mermaid）
06_dev_notes.md       # Step 7: 开发记录（关键决策/坑点）
07_review.md          # Step 8: 代码评审记录
08_release.md         # Step 9: 上线记录
_state.md             # 进度状态（每步完成后更新）
```

规范文件（所有需求共享）：
```
~/.claude/requirements/_specs/flowchart-spec.md
~/.claude/requirements/_specs/classdiagram-spec.md
```

## 命令：`/req init <id> <需求名称>`

**作用**：初始化一个新需求，创建目录和 `_state.md`

**执行步骤**：
1. 在 `~/.claude/requirements/` 下创建目录 `{id}-{name}/`
2. 创建 `_state.md`，内容如下：

```markdown
# 需求：{需求名称}
**ID**：{id}
**创建时间**：{当前日期}
**当前阶段**：未开始

## 进度
- [ ] Step 1: 背景信息 (`/req background`)
- [ ] Step 2: 变更点 (`/req changes`)
- [ ] Step 3: 代码调研 (`/req codebase`)
- [ ] Step 4: 方案设计 (`/req solutions`)
- [ ] Step 5: 流程图 (`/req flowchart`)
- [ ] Step 6: 类图 (`/req classdiagram`)
- [ ] Step 7: 开发记录 (`/req dev`)
- [ ] Step 8: 代码评审 (`/req review`)
- [ ] Step 9: 上线 (`/req release`)

## 关键约束（在此记录技术限制/业务规则）
（待填写）

## 选定方案
（待选定）
```

3. 输出提示：「需求 [{id}] {name} 已初始化，执行 `/req background` 开始第一步」
4. 将此需求设为当前激活需求（在 `~/.claude/requirements/_active` 文件中写入需求目录路径）

## 命令：`/req background`

**作用**：Step 1 —— 梳理需求背景信息

**加载的上下文**：
- `_state.md`（当前进度）

**执行流程**：

1. 读取 `_state.md`，确认当前激活需求
2. 进入背景梳理模式，引导用户提供以下信息（逐项确认，不要一次问完）：
   - **业务背景**：这个需求解决什么问题？用户/业务方的诉求是什么？
   - **技术背景**：当前系统的相关现状（已有能力、已知限制）
   - **上下游依赖**：涉及哪些系统/服务/团队？
   - **关键约束**：不可违反的技术限制、业务规则、时间节点
   - **成功标准**：如何判断这个需求做完了、做对了？

3. 信息收集完成后，整理成结构化文档，**预览给用户确认**
4. 用户确认后，写入 `00_background.md`
5. 更新 `_state.md`：Step 1 打勾，当前阶段改为「Step 1 已完成」
6. 提示：「背景信息已保存，执行 `/req changes` 进入下一步」

**`00_background.md` 格式**：
```markdown
# 背景信息
**最后更新**：{日期}

## 业务背景
{内容}

## 技术背景
{内容}

## 上下游依赖
| 系统/服务 | 角色 | 交互方式 |
|----------|------|---------|
| ...      | ...  | ...     |

## 关键约束
- {约束1}
- {约束2}

## 成功标准
- {标准1}
- {标准2}
```

## 命令：`/req changes`

**作用**：Step 2 —— 整理需求变更点

**加载的上下文**：
- `00_background.md`
- `_state.md`

**执行流程**：

1. 读取以上文件，输出一句话背景摘要，让用户确认上下文正确
2. 引导用户粘贴或描述变更点（PRD 片段、需求文档、口头描述均可）
3. 将变更点整理为标准格式，区分：
   - **新增功能**（需要从无到有实现的）
   - **修改功能**（改变已有行为的）
   - **删除功能**（下线/废弃的）
   - **非功能性变更**（性能、安全、配置变更等）
4. 对每个变更点标注**影响范围**（估计涉及哪些模块/服务）
5. **预览给用户确认**，可修改
6. 写入 `01_changes.md`，更新 `_state.md`

**`01_changes.md` 格式**：
```markdown
# 变更点
**最后更新**：{日期}

## 新增功能
- [ ] {变更描述} | 影响范围：{模块}

## 修改功能
- [ ] {变更描述} | 影响范围：{模块} | 原行为：{xxx} → 新行为：{xxx}

## 删除功能
- [ ] {变更描述}

## 非功能性变更
- [ ] {变更描述}

## 变更范围评估
预计涉及模块：{模块列表}
预计改动规模：小/中/大
```

## 命令：`/req codebase`

**作用**：Step 3 —— 调研现有代码实现

**加载的上下文**：
- `00_background.md`（了解业务范围）
- `01_changes.md`（知道要改哪些模块）
- `_state.md`

**执行流程**：

1. 读取变更点中的影响范围，列出需要调研的模块清单
2. 逐模块引导用户（或自己通过工具）分析：
   - 核心类/接口在哪里？
   - 数据流是怎样的？
   - 有哪些扩展点/限制点？
   - 有无历史遗留问题需要注意？
3. 如果用户在代码仓库中，主动使用 Bash/Read 工具搜索相关代码
4. 整理调研结论，**重点标注对方案设计有影响的发现**
5. 写入 `02_codebase.md`，更新 `_state.md`

**`02_codebase.md` 格式**：
```markdown
# 现有代码调研
**最后更新**：{日期}

## 模块：{模块名}
**核心文件**：
- `{文件路径}:{行号}` - {说明}

**当前实现方式**：
{描述}

**对方案设计的影响**：
- ⚠️ {限制/注意点}
- ✅ {可复用的能力}

---

## 关键发现汇总
| 发现 | 影响 | 建议 |
|------|------|------|
| ...  | ...  | ...  |
```

## 命令：`/req solutions`

**作用**：Step 4 —— 设计并评估多个实现方案

**加载的上下文**：
- `00_background.md`
- `01_changes.md`
- `02_codebase.md`
- `_state.md`

**执行流程**：

1. 综合前三步信息，提出 1~3 个可行方案
2. 每个方案必须包含：
   - **方案概述**（一句话描述核心思路）
   - **实现路径**（关键步骤，不超过 5 条）
   - **可行性**：技术上能否实现？有无先决条件？
   - **开发难度**：低/中/高，说明原因
   - **风险点**：列出 top 3 风险，每个风险附应对措施
   - **与现有代码的兼容性**：改动量大小、是否破坏已有逻辑
3. 生成**方案对比矩阵**
4. 给出推荐方案及理由，**等待用户选择**
5. 用户选定后，在 `_state.md` 的「选定方案」中记录
6. 写入 `03_solutions.md`，更新 `_state.md`

**`03_solutions.md` 格式**：
```markdown
# 方案设计
**最后更新**：{日期}
**选定方案**：{方案X}（待选定 / 方案一 / 方案二 / 方案三）

## 方案一：{名称}

### 概述
{一句话描述}

### 实现路径
1. {步骤1}
2. {步骤2}

### 评估
| 维度 | 评分 | 说明 |
|------|------|------|
| 可行性 | ⭐⭐⭐⭐⭐ | {说明} |
| 开发难度 | 低/中/高 | {说明} |
| 风险 | 低/中/高 | {说明} |
| 兼容性 | 好/一般/差 | {说明} |

### 风险点
1. **{风险}**：{应对措施}
2. **{风险}**：{应对措施}

---

## 方案对比矩阵
| 方案 | 可行性 | 难度 | 风险 | 兼容性 | 推荐度 |
|------|--------|------|------|--------|--------|
| 方案一 | ... | ... | ... | ... | ⭐⭐⭐ |

## 推荐：方案X
{推荐理由}
```

## 命令：`/req flowchart`

**作用**：Step 5 —— 绘制执行流程图

**加载的上下文**：
- `00_background.md`（业务背景）
- `01_changes.md`（变更范围）
- `03_solutions.md`（选定方案的实现路径）
- `~/.claude/requirements/_specs/flowchart-spec.md`（流程图规范）

**执行流程**：

1. 读取选定方案的实现路径，理解核心流程
2. 严格按照 `flowchart-spec.md` 中的规范绘制 Mermaid 流程图
3. **以代码块形式预览**，供用户审查
4. 等待用户反馈（可多轮修改，直到满意）
5. 用户确认后写入 `04_flowchart.mmd`，更新 `_state.md`
6. **重要**：每次修改都更新文件，不要等到"完全满意"才保存

---

## 命令：`/req classdiagram`

**作用**：Step 6 —— 绘制核心类图

**加载的上下文**：
- `00_background.md`
- `03_solutions.md`（选定方案）
- `04_flowchart.mmd`（流程图，了解数据流）
- `02_codebase.md`（现有类结构参考）
- `~/.claude/requirements/_specs/classdiagram-spec.md`（类图规范）

**执行流程**：

1. 从流程图和方案中识别核心领域对象
2. 确认需要画的类的范围（新增类 + 修改类 + 关键依赖类）
3. 严格按照 `classdiagram-spec.md` 规范绘制 Mermaid 类图
4. **预览给用户**，等待反馈
5. 可按模块/层次分多张图（如：领域层、服务层单独画）
6. 用户确认后写入 `05_classdiagram.mmd`，更新 `_state.md`

---

## 命令：`/req dev`

**作用**：Step 7 —— 进入开发辅助模式，记录开发过程中的关键决策

**加载的上下文**：
- `00_background.md`（背景）
- `01_changes.md`（变更清单，作为 checklist）
- `03_solutions.md`（选定方案）
- `04_flowchart.mmd`（流程图）
- `05_classdiagram.mmd`（类图）

**执行流程**：

1. 展示变更清单（来自 `01_changes.md`），作为开发 checklist
2. 进入开发辅助模式：
   - 回答代码实现问题时，始终参考已有的流程图和类图
   - 如设计有偏差，主动提醒并建议更新图纸
3. 每当用户做出与原方案不同的决策时，提示记录到 `06_dev_notes.md`
4. 命令 `/req dev save` 手动触发保存当前开发记录

**`06_dev_notes.md` 格式**：
```markdown
# 开发记录
**最后更新**：{日期}

## 变更清单进度
- [x] {变更点1}（完成时间：{日期}）
- [ ] {变更点2}

## 关键决策记录
### {日期} - {决策描述}
**背景**：{为什么要做这个决策}
**决策**：{做了什么}
**影响**：{对其他模块/后续开发的影响}

## 遗留问题
- [ ] {问题描述}（优先级：高/中/低）

## 与原方案的偏差
- {原方案说xxx，实际实现是xxx，原因是xxx}
```

## 命令：`/req review`

**作用**：Step 8 —— 代码评审辅助

**加载的上下文**：
- `03_solutions.md`（选定方案，作为评审基准）
- `04_flowchart.mmd`（流程图，验证代码是否符合设计）
- `05_classdiagram.mmd`（类图，验证类设计）
- `06_dev_notes.md`（开发记录，了解偏差背景）

**执行流程**：

1. 展示需求设计文档摘要（方案+流程图+类图），确认评审基准
2. 调用 `/code-review` skill 进行代码质量检查
3. 在代码质量问题之外，额外检查：
   - 实现是否符合选定方案？
   - 代码流程是否与流程图一致？
   - 类设计是否与类图吻合？
   - `06_dev_notes.md` 中记录的偏差是否有合理解释？
4. 将评审结论（问题列表 + 是否通过）写入 `07_review.md`
5. 更新 `_state.md`

## 命令：`/req release`

**作用**：Step 9 —— 上线后更新需求进度

**加载的上下文**：
- `_state.md`
- `01_changes.md`（变更清单，核对上线了哪些）

**执行流程**：

1. 引导用户确认以下信息：
   - 上线时间
   - 上线的变更点（哪些上线了，哪些没有）
   - 上线结果（成功/部分成功/回滚）
   - 上线后观察到的问题
2. 写入 `08_release.md`
3. 将 `_state.md` 中已上线的变更点打勾
4. 如全部完成，将需求状态标为「✅ 已上线」

**`08_release.md` 格式**：
```markdown
# 上线记录

## 上线信息
**上线时间**：{日期时间}
**上线结果**：成功 / 部分成功 / 回滚

## 上线变更清单
- [x] {变更点1} - 已上线
- [ ] {变更点2} - 未上线（原因：{xxx}）

## 上线后观察
{上线后的现象、监控数据、异常情况}

## 遗留事项
- {还未完成的事项，下一期处理}
```

## 命令：`/req status`

**作用**：查看当前需求的整体进度

**执行**：读取 `_state.md`，以清晰的格式展示当前进度、各文件状态、关键约束和选定方案。

---

## 命令：`/req load <req-id>`

**作用**：切换到指定需求，恢复上下文

**执行**：
1. 找到 `~/.claude/requirements/` 下匹配 `{req-id}-*` 的目录
2. 读取 `_state.md`，输出「需求恢复简报」：
   - 需求名称和背景（一句话）
   - 当前进度（到哪一步了）
   - 选定方案（一句话）
   - 上次未完成的事项
3. 更新 `~/.claude/requirements/_active` 为此需求路径
4. 提示用户下一步操作

---

## 命令：`/req list`

**作用**：列出所有需求及其进度

**执行**：扫描 `~/.claude/requirements/` 下的所有需求目录，读取各自的 `_state.md`，以表格输出：

```
| ID | 需求名称 | 当前阶段 | 最后更新 | 状态 |
|----|---------|---------|---------|------|
```

---

## 错误处理

- 如找不到当前激活需求，提示用户执行 `/req load <id>` 或 `/req init <id> <name>`
- 如某步骤的依赖文件不存在，提示用户先完成前置步骤
- 如用户跳步（如直接 `/req flowchart` 但 `03_solutions.md` 不存在），询问是否确认跳过，并记录在 `_state.md` 中
