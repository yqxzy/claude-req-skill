## 1. 背景与动机

### 1.1 起点

在日常需求开发中，开发者越来越多地借助 Claude Code 辅助完成各类任务：梳理背景、整理变更、调研代码、设计方案、画流程图、画类图、辅助编码……

然而随着使用深入，一个核心矛盾逐渐显现：

> **Claude 的单次对话上下文是有限且线性的。** 当一个对话承载了需求开发全流程的所有信息，上下文会变得极长，Claude 的生成质量会明显下降——回答开始变得泛化、忽略细节、前后矛盾。

### 1.2 现有做法的问题

| 做法 | 问题 |
|------|------|
| 一个对话搞定所有事 | 上下文过长，生成质量下降；多轮修改后容易丢失早期决策 |
| 每个任务开新对话 | 每次都要重新描述背景，效率极低；容易遗漏上次的结论 |
| 手动复制粘贴上下文 | 繁琐、不一致、缺乏结构；没有标准格式，难以复用 |
| 依赖记忆/笔记 | 依赖个人习惯，团队不一致；上下文不完整时生成质量仍差 |

### 1.3 目标

设计一套 **Claude Code Skill**，让需求开发流程：

- **可拆分**：不同阶段可以在不同对话中独立完成
- **可恢复**：任何时候开启新对话，都能在 6 行简报内完整恢复上下文
- **可并行**：同时进行多个需求时，对话之间互不干扰
- **自动联动**：需求变更时，已完成的工作项能自动评估影响并更新

## 2. 问题定义

### 2.1 核心痛点

**P1 — 上下文膨胀导致质量下降**

需求开发的典型对话轨迹：
```
背景介绍(2000 tokens) → 变更点讨论(1500) → 代码调研(3000) 
→ 方案设计(2000) → 流程图修改×3(3000) → 类图讨论(2000) 
→ 编码问题(持续增长...)
```
超过约 50k tokens 后，Claude 对早期信息的注意力明显衰减。

**P2 — 跨对话上下文断裂**

重新开启对话时，开发者需要手动重建上下文。如果不重建，Claude 的回答缺乏需求背景，生成物与已有设计不一致。

**P3 — 多需求并行混乱**

同时开发多个需求时，对话之间容易串数据，或者需要频繁切换上下文，效率损耗严重。

**P4 — 需求变更的涟漪效应**

需求变更后，开发者需要手动判断哪些已完成的工作需要更新（测试 Case、流程图、类图……），容易遗漏，导致各工作项之间不一致。

### 2.2 非目标

- 不替代正式的项目管理工具（Jira、飞书多维表格等）
- 不做团队协作/多人共享（当前是个人开发者工具）
- 不做代码自动提交/CI 集成

## 3. 用户场景

### 场景 A：单需求全流程

> 开发者独立完成一个中等规模需求（预计 3～5 天），全程使用 Skill 辅助。

```
Day 1 - 对话①：
  /req init REQ-101 用户积分兑换
  /req background    → 梳理业务背景
  /req changes       → 整理 PRD 变更点
  /req codebase      → 调研积分模块现有代码

Day 2 - 对话②（新对话，接续）：
  /req load REQ-101  → 6 行简报恢复上下文
  /req solutions     → 设计并选定实现方案
  /req testcases     → 生成测试 Case 表格
  /req flowchart     → 画执行流程图（多轮修改）
  /req classdiagram  → 画核心类图

Day 3-4 - 对话③（开发阶段）：
  /req load REQ-101
  /req dev           → 开发辅助，遇到问题随时参考流程图/类图

Day 5 - 对话④（评审+上线）：
  /req load REQ-101
  /req review        → 代码评审
  /req release       → 记录上线结果
```

### 场景 B：多需求并行

> 开发者同时跟进 REQ-101 和 REQ-102，两个需求交替推进。

```
对话①：/req load REQ-101 → 专注 REQ-101 的流程图设计
对话②：/req load REQ-102 → 专注 REQ-102 的代码调研
对话③：/req load REQ-101 → 继续修改流程图
```

两个对话完全隔离，`REQ-101` 的上下文不会污染 `REQ-102`，反之亦然。

### 场景 C：需求迭代（需求变更）

> 需求开发到一半，产品说「把 A 功能改为 B」。

```
开发者（在任意对话中输入）：
  「把积分过期时间改为按自然月，不再按购买日期计算」

Skill 自动检测到变更信号：
  ① 更新 01_changes.md
  ② 评估影响：testcases 需要更新，flowchart 需要更新，classdiagram 不受影响
  ③ 自动更新 04_testcases.md（追加新 Case）
  ④ 自动更新 05_flowchart.puml（修改时间判断逻辑分支）
  ⑤ 输出更新报告
```

### 场景 D：需求恢复（隔了一段时间重新捡起来）

> 某需求搁置了一周，重新开始。

```
/req load REQ-101

输出：
  📋 [REQ-101] 用户积分兑换  路径：~/.claude/requirements/REQ-101.../
  背景：用户通过积分兑换商品，支持跨店铺兑换
  进度：已完成 Step 1-5，当前在 Step 6 流程图
  选定方案：方案二 - 异步扣减 + 幂等校验
  约束：不能修改 points_ledger 已有表结构
  上次停留：流程图第二轮修改后保存，待完成类图
  建议下一步：/req classdiagram
```

## 4. 功能设计

### 4.1 10 步开发流程

| 步骤 | 命令 | 输入 | 产出文件 |
|------|------|------|---------|
| Step 1 | `/req background` | 用户口述 | `00_background.md` |
| Step 2 | `/req changes` | PRD/口述（业务行为） | `01_changes.md` |
| Step 3 | `/req codebase` | 代码库（自动搜索） | `02_codebase.md` |
| Step 4 | `/req solutions` | 前三步产出 | `03_solutions.md` |
| Step 5 | `/req testcases` | 变更点 + 选定方案 | `04_testcases.md` |
| Step 6 | `/req flowchart` | 选定方案 + 流程图规范 | `05_flowchart.puml` |
| Step 7 | `/req classdiagram` | 流程图 + 代码调研 + 类图规范 | `06_classdiagram.puml` |
| Step 8 | `/req dev` | 变更清单 + 流程图 + 类图 | `07_dev_notes.md` |
| Step 9 | `/req review` | 流程图 + 类图 + 开发记录 | `08_review.md` |
| Step 10 | `/req release` | 变更清单 | `09_release.md` |

### 4.2 变更联动更新

检测到以下信号词自动触发，无需手动命令：

```
「新增变更」「追加变更」「改为」「变更为」「修改成」「调整为」「删除变更」「去掉这个」
```

**联动规则**：

| 已完成步骤 | 处理方式 | 原因 |
|-----------|---------|------|
| Step 2 变更点 | 自动更新 | 必须更新，是所有后续步骤的基础 |
| Step 5 测试 Case | 自动追加 | 新变更大概率需要新 Case，追加安全 |
| Step 6 流程图 | 自动更新 | 流程图结构化，可精确修改受影响部分 |
| Step 7 类图 | 自动更新 | 同上 |
| Step 3 代码调研 | 标注复查 | 需要重新搜索代码，自动更新不可靠 |
| Step 4 方案设计 | 标注复查 | 方案评估逻辑复杂，需人工判断 |

### 4.3 图表规范

**流程图**（PlantUML Sequence Diagram）：
- 一个 `box` = 一个服务，box 名称 = 服务名
- 分支用 `alt/else`，降级路径 note 标红 `#FFE4E1`
- 阶段用 `== 标题 ==` 分隔

**类图**（PlantUML Class Diagram）：
- 只画 Repository 层及以上
- 修改类：`#lightblue`，新增类：`#lightgreen`，未变动：无色
- 新增方法用 `//` 注释说明用途
- 图后附变更说明文字

### 4.4 测试 Case 格式

```markdown
| TC# | 测试场景 | 前置条件 | 操作 | 期望结果 |
|-----|---------|---------|------|---------|
```

覆盖维度：主流程 / 开关配置分支 / 降级场景 / 边界条件 / 回归场景

## 5. 架构设计

### 5.1 整体结构

```
Claude Code Skill 调用链：

用户输入 /req <命令>
    ↓
SKILL.md（薄路由 + 变更探测）
    ↓ 路由到对应文件
cmd/<命令>.md（命令级指令，14～30 行）
    ↓ 声明加载哪些文件
需求目录文件（_state.md / 0X_xxx.md）+ 规范文件
    ↓ Claude 执行后
写回需求目录文件
```

### 5.2 文件分层

```
~/.claude/
├── skills/req/                    # Skill 层（指令）
│   ├── SKILL.md                   # 路由 + 变更探测
│   └── cmd/                       # 命令实现（14～30 行/文件）
│       ├── load.md                # 核心：跨对话入口
│       ├── propagate.md           # 变更联动逻辑
│       └── [background～release].md
│
└── requirements/
    ├── _specs/                    # 规范层（共享，所有需求通用）
    │   ├── flowchart-spec.md      # PlantUML 时序图规范
    │   └── classdiagram-spec.md   # PlantUML 类图规范
    │
    └── {id}-{name}/               # 数据层（每需求独立）
        ├── _state.md              # 进度快照（跨对话的锚点）
        └── 00～09_*.md/.puml      # 各步骤产出文件
```

### 5.3 上下文加载策略

每个命令只加载自己需要的文件，避免无关内容占用上下文：

| 命令 | 加载文件 |
|------|---------|
| background | `_state.md` |
| changes | `_state.md` + `00_background.md` |
| codebase | `_state.md` + `01_changes.md`（只读影响模块） |
| solutions | `_state.md` + `00` + `01` + `02` |
| testcases | `_state.md` + `01_changes.md` + `03_solutions.md`（只读选定方案） |
| flowchart | `_state.md` + `03_solutions.md`（只读实现路径） + 流程图规范 |
| classdiagram | `_state.md` + `05_flowchart.puml` + `02_codebase.md`（只读核心类） + 类图规范 |
| dev | `_state.md` + `01_changes.md` + `05_flowchart.puml` + `06_classdiagram.puml` |
| review | `_state.md` + `05` + `06` + `07_dev_notes.md` |

### 5.4 跨对话隔离机制

**问题**：多个对话同时运行，如何知道当前对话在处理哪个需求？

**方案**：不使用任何共享状态文件（去除了 `_active` 文件方案）。

```
对话 A 执行 /req load REQ-101
→ 读取 REQ-101 的 _state.md，输出简报，路径写入对话记忆
→ 后续所有命令从对话记忆中取路径

对话 B 执行 /req load REQ-102
→ 独立读取 REQ-102，与对话 A 完全无关
→ 两个对话的"当前需求"存在各自的上下文中，不共享任何文件
```

**`_state.md` 设计**（进度快照，跨对话的锚点）：

```markdown
需求：{名称} | ID：{id} | 创建：{日期}
背景摘要：{20字以内}
选定方案：{方案名} - {一句话描述} / 未选定
关键约束：{逗号分隔}

进度：
[x] Step1-背景  [x] Step2-变更  [x] Step3-调研
[x] Step4-方案  [ ] Step5-测试Case
[ ] Step6-流程图 ...

上次停留：{描述当前进展和建议下一步}
```

## 6. 关键设计决策

### 决策 1：Skill 而非 Workflow

**背景**：最初考虑用 Claude Code Workflow（多 Agent 编排）实现。

**选择 Skill 的原因**：
- 需求开发的核心操作是**对话式迭代**（流程图改了又改、方案选了又调整），这是 Workflow 的弱项——Workflow 子 Agent 是 fire-and-forget，不支持中途插入对话
- Skill 是用户主动触发的，契合「每步由用户决定何时执行」的需求
- Skill + 文件 I/O 已经能解决上下文管理问题，不需要 Workflow 的并发能力

**Workflow 保留空间**：部分步骤（代码调研、方案设计、代码评审）在未来可以用 Workflow 做并行分析（见第 8 节）。

### 决策 2：薄路由 + cmd 分文件

**背景**：初版把所有命令的指令写在一个 474 行的 SKILL.md 里。

**问题**：每次执行任意命令都加载全部 474 行，即使只用 `/req background`，也加载了 `/req release` 的全部指令。

**现方案**：
- `SKILL.md` 只保留路由表（23 行）
- 每个命令的指令独立存储在 `cmd/xxx.md`（14～30 行）
- 执行时只加载当前命令的文件

**效果**：单次命令的 Skill 指令加载量从 474 行降至约 40 行。

### 决策 3：对话记忆替代 `_active` 文件

**背景**：最初用 `~/.claude/requirements/_active` 文件记录当前激活需求的路径。

**问题**：`_active` 是全局共享文件。多需求并行时，对话 B 的 `/req load REQ-002` 会覆盖对话 A 写入的路径，导致对话 A 后续操作串到 REQ-002。

**现方案**：`/req load` 把需求路径输出到对话，后续命令从对话记忆中取路径，不读写任何共享文件。

**代价**：每次新对话必须先执行 `/req load`，不能直接上来就 `/req flowchart`。这是可接受的，且有明确的报错提示引导用户。

### 决策 4：先写再改，不预览确认

**背景**：初版流程是「整理内容 → 预览给用户确认 → 用户说可以 → 写文件」。

**问题**：多了一次无效的交互轮次。实际上用户看到产出后才会提意见，确认环节纯属冗余。

**现方案**：执行完直接写文件，然后展示内容，等修改意见。

### 决策 5：`changes` 只记录业务行为

**背景**：初版变更点格式包含「影响模块」「技术分类」等字段。

**问题**：变更点阶段（Step 2）不应该预设技术实现——那是 Step 3（代码调研）和 Step 4（方案设计）的工作。过早引入技术视角会污染变更描述的纯粹性，且影响模块的判断往往不准确。

**现方案**：变更点只描述「什么功能/行为发生了什么变化」，不涉及模块、类、接口等技术概念。

### 决策 6：变更传播的自动/手动边界

**自动更新**：testcases（追加安全）、flowchart/classdiagram（结构化，可精确修改）

**标注复查**：codebase（需要重新搜索代码）、solutions（方案评估逻辑复杂）

**原则**：对「追加操作安全、修改范围可预估」的产出物做自动更新；对「需要人工判断、修改错了影响大」的产出物只标注提醒。

## 7. 当前限制

### 7.1 Claude 对话记忆的可靠性

当前需求的路径依赖 Claude 的对话内记忆。在**极长对话**（超过 100k tokens）中，早期 `/req load` 的输出可能因上下文压缩而变得不清晰，影响后续命令的路径识别。

**现有缓解**：`/req load` 输出的路径是明确的完整路径字符串，在被引用时有很强的特征，不容易被混淆。

**未来方案**：见 8.3。

### 7.2 变更探测的召回率

当前依赖关键词匹配，对于隐含的变更描述（「我们决定不做 A，改做 B」）可能无法触发。同时，误触发的可能性也存在。

### 7.3 单 Agent，不并行

代码调研（Step 3）、方案设计（Step 4）、代码评审（Step 9）等步骤当前是串行的单 Agent 处理，对于复杂需求效率较低。

### 7.4 无法感知代码变化

`02_codebase.md` 记录的是某一时刻的代码调研结果。如果代码库在需求开发期间发生了变化（比如其他需求上线了），文件内容会过时，但 Skill 目前不感知这种变化。

## 8. 后续迭代规划

### 8.1 P0 — 稳定性与易用性

**子命令补全**：
- `/req dev note` — 快速记录开发决策（已实现），增加 `/req dev checklist` 展示当前变更完成进度
- `/req undo` — 撤销上一步的文件写入（读取备份）

**`_state.md` 自动备份**：每次写入前保留上一版本为 `_state.md.bak`，支持基础的回滚。

**更好的错误提示**：当前如果找不到需求目录，报错信息较简单；改为展示 `/req list` 的内容辅助用户定位。

---

### 8.2 P1 — Workflow 赋能的并行分析

部分步骤适合用 Claude Code Workflow 做并行分析，对话保持串行（用户交互），但分析阶段在后台并发执行：

**`/req codebase` 并行化**：
```javascript
// 识别涉及的模块列表后，并行分析各模块
const modules = await agent('从变更点识别需要调研的模块列表')
const results = await pipeline(
  modules.list,
  module => agent(`分析 ${module} 模块的现有实现`)
)
```
预期效果：5 个模块的调研从串行 5 分钟缩短为并行约 1～2 分钟。

**`/req solutions` 并行化**：
```javascript
// 多个 Agent 各自独立设计一个方案，再汇总对比
const solutions = await parallel([
  () => agent('从可行性优先角度设计方案'),
  () => agent('从最小改动原则设计方案'),
  () => agent('从未来扩展性优先设计方案'),
])
```
预期效果：方案多样性更高，对比更充分。

**`/req review` 多维并行**：
```javascript
const reviews = await parallel([
  () => agent('从代码质量维度评审'),
  () => agent('从与设计文档一致性维度评审'),
  () => agent('从安全和边界处理维度评审'),
])
```

### 8.3 P1 — 更健壮的需求上下文持久化

**问题**：当前路径依赖对话记忆，超长对话可能丢失。

**方案**：Session 级别的状态文件。

在 `/req load` 时，除了输出简报，还将需求路径写入一个 **会话级** 文件（基于时间戳命名，区分不同对话实例）：

```
~/.claude/requirements/_sessions/
  session-20260630-143022.json   ← 对话 A 的状态
  session-20260630-150155.json   ← 对话 B 的状态
```

每个 session 文件记录：`{ requirementId, requirementPath, loadedAt }`

SKILL.md 在执行命令时优先从对话记忆取路径，失败时 fallback 到读取最近的 session 文件。

### 8.4 P2 — 需求模板自定义

不同类型的需求（功能需求、性能优化、重构、Bug 修复）适合的步骤不同。

**目标**：支持按需求类型自定义流程：

```
/req init REQ-201 缓存优化 --type=perf

perf 模板的步骤：
  Step 1: background
  Step 2: changes  
  Step 3: codebase（重点：现有性能瓶颈分析）
  Step 4: benchmark（性能基准测试方案）← 新增
  Step 5: solutions
  Step 6: flowchart
  ...
```

### 8.5 P2 — 团队共享规范

当前规范文件（flowchart-spec.md / classdiagram-spec.md）存在个人目录，无法团队共享。

**目标**：支持从项目级目录加载规范，优先级高于全局规范：

```
项目根目录/.claude/req-specs/
  flowchart-spec.md    ← 团队定制规范，覆盖全局
  classdiagram-spec.md
```

**场景**：团队对流程图有统一的 skinparam 要求、特定的颜色方案，可以统一在项目级规范中配置，所有成员自动生效。

---

### 8.6 P3 — 与代码库的双向感知

**目标**：当代码库发生变化时，Skill 能感知到哪些已完成的工作项可能已过时。

**方案**：
- `02_codebase.md` 记录调研时涉及的文件路径和最后修改时间
- `/req dev` 开始时，检查这些文件是否在调研后被修改过
- 如有变化，提示「以下文件自上次调研后已修改，建议重新确认」
